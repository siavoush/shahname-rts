# tools/L7_allowlist.txt — sanctioned exceptions to lint rule L7
# (STUDIO_PROCESS §9.M7 defensive-fallback-masking).
#
# One entry per line, format:
#
#   <repo-relative-file>:<line-context-substring> — <one-line WHY>
#
# Matching is mechanical (tools/lint_simulation.sh L7): a production hit is
# allowlisted when its file equals <repo-relative-file> (or the entry file is
# `*` — a CATEGORY entry matching any file under game/scripts/) AND the hit's
# source line contains <line-context-substring> verbatim. Keep the context
# substring tight — usually the full probe expression `has_method(&"x")`.
# The ` — ` (space-emdash-space) separator between context and WHY is
# load-bearing for the parser; don't replace it with a hyphen.
#
# NOT needed in this file (exempt in the lint itself):
#   - comment lines mentioning has_method/has_signal/has_property
#   - hard-asserts: assert(x.has_method(&"y"), "...") — the §9.M7 RIGHT pattern
#   - anything under game/tests/ (out of scope entirely)
#
# Lines starting with `#` and blank lines are ignored.
# Owner: qa-engineer. Every entry needs a WHY. Prune entries whose call sites
# are gone at the next L7 sweep; new guards need a new entry + reviewer signoff.
