# Architecture — Target Shape and Build State

> This document is the **orientation layer** for the project. After `MANIFESTO.md` and `CLAUDE.md`, this is the first thing anyone (agent or human) reads when picking up work — especially after a context boundary (compaction, time off, fresh session). It exists so you don't have to re-read every contract and every commit just to know where you are.
>
> **What lives here:** the target shape, the build state, the data flow, the directory map. **What lives elsewhere:** the *decisions* that shaped each subsystem (in `docs/*_CONTRACT.md`), the *plan* for getting there (in `02_IMPLEMENTATION_PLAN.md`), the *principles* behind everything (in `MANIFESTO.md`).

## 1. The System Map

The game is a deterministic real-time simulation with seven phases per tick at 30 Hz. Subsystems are organized into four layers:

```
┌─────────────────────────────────────────────────────────────────┐
│                       UI LAYER (off-tick)                        │
│  Camera • SelectionManager • CommandManager • HUD • Minimap     │
│  DebugOverlayManager (F1-F4) • Farr Gauge • Build Menu          │
└────────────────────┬────────────────────────────────────────────┘
                     │ reads sim state freely
                     │ writes sim state ONLY through queued commands
                     ▼
┌─────────────────────────────────────────────────────────────────┐
│                   SIMULATION LAYER (on-tick)                     │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ SimClock (30 Hz, runs the phase pipeline)                │   │
│  │   input → ai → movement → spatial_rebuild → combat       │   │
│  │            → farr → cleanup                              │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  Systems (phase coordinators):                                   │
│   - InputSystem             - CombatSystem                       │
│   - AIControllerHost        - FarrSystem                         │
│   - MovementSystem          - CleanupSystem                      │
│   - SpatialIndex (rebuild)                                       │
│                                                                  │
│  Components (extend SimNode, register with phase coordinators):  │
│   - HealthComponent  - MovementComponent  - CombatComponent      │
│   - SelectableComponent  - SpatialAgentComponent                 │
│                                                                  │
│  Entities (composed of components):                              │
│   - Units (Kargar, Piyade, Kamandar, Savar, Asb-savar, Rostam)   │
│   - Buildings (Throne, Khaneh, Mazra'eh, Atashkadeh, ...)        │
│   - ResourceNode → MineNode | Mazra'eh                           │
└────────────────────┬────────────────────────────────────────────┘
                     │ emits typed signals (write-shaped + read-shaped)
                     ▼
┌─────────────────────────────────────────────────────────────────┐
│                          EVENT BUS                               │
│   Single autoload. Every signal declared with typed args.        │
│   Telemetry sink subscribes to write-shaped signals.             │
└────────────────────┬────────────────────────────────────────────┘
                     │
       ┌─────────────┴─────────────┐
       ▼                           ▼
┌──────────────┐          ┌──────────────────┐
│ Telemetry    │          │ MatchLogger      │
│ (Phase 0)    │          │ (Phase 6)        │
│ in-memory    │          │ NDJSON to disk   │
│ JSON dump    │          │ per tick events  │
└──────────────┘          └──────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                       FOUNDATION LAYER                           │
│  SimClock • TimeProvider • GameRNG (domain-keyed) • EventBus    │
│  Constants • BalanceData (resource) • GameState                  │
│  SimNode (base class) • StateMachine + State (core)              │
│  IPathScheduler (interface) → NavigationAgent | MockPathScheduler│
└─────────────────────────────────────────────────────────────────┘
```

**Key invariant:** gameplay state mutates only inside `_sim_tick()` calls dispatched by `SimClock`. UI reads freely; UI never writes synchronously in a sink callback (uses queue-then-drain). Per Sim Contract §1.

### v0.2.0 — Phase 0 Session 1 (2026-05-01)

- **EventBus.connect_sink — implementation diverged from the contract sketch.** Sim Contract §7 originally sketched `func connect_sink(callable): for sig in _SINK_SIGNALS: get(sig).connect(func(...args): callable.call(sig, args))`. GDScript does **not** support varargs lambdas (`func(...args)` is not valid syntax), so the actual implementation registers one hand-rolled per-signal forwarder Callable per `(sink, signal)` pair, dispatching to `sink.call(signal_name, args_array)`. The public API (`connect_sink(sink: Callable)` / `disconnect_sink(sink: Callable)` taking a sink that receives `(StringName, Array)`) is unchanged from the contract; only the internal forwarder construction differs. ### v0.10.0 — Phase 1 Session 1 wave 2 (2026-05-01) — ui-developer (selection)

### v0.13.0 — Phase 1 Session 1 wave 2 (2026-05-01) — ai-engineer (Idle / Moving states)

- **`UnitState_Idle` and `UnitState_Moving` registered by the `Unit` base class, not by concrete subclasses.** State Machine Contract §3.1 says "Every State instance for a unit is allocated at spawn (in the unit's `_ready`)." The kickoff doc proposed concrete `Kargar`-style subclasses register their own states; the base class registers Idle and Moving (the universally-useful pair) so every concrete unit type ships with a working FSM out of the box. Subclasses that want extra states (Gathering for workers, Attacking for combatants, etc.) call `fsm.register(...)` BEFORE chaining to `super._ready()`. The base's `_ready` then idempotently registers Idle/Moving (only if not already present) and calls `init(&"idle")` only if `current` is still null. Rationale: this minimizes the boilerplate concrete subclasses repeat (gameplay-systems' `Kargar` doesn't have to know about Idle/Moving) and matches engine-architect's wave-1 retro note "If [Unit] does [register stub states], your job is just to make the actual UnitState subclasses exist and be findable."

- **`UnitState_Idle` / `UnitState_Moving` extend `unit_state.gd` by path-string, NOT by `class_name UnitState`.** Same project-wide class_name registry race (§6 v0.4.0, v0.9.0). Initial `extends UnitState` triggered the cascade where unit.gd's `class_name Unit` failed to register, surfacing as `Invalid call. Nonexistent function 'reset_id_counter (via call)' in base 'GDScript'` and `Invalid assignment of property or key 'unit_type' on a base object of type 'CharacterBody3D'` in test fixtures. Path-string base + `class_name UnitState_Idle` / `class_name UnitState_Moving` retained for typed external references. Unit also references the state scripts via path-string preload (`const _UnitStateIdleScript: Script = preload(...)`) instead of by class_name to dodge the same race when unit.gd is loaded in test contexts.

- **`StateMachine.transition_to_next` stashes the dispatched Command's kind/payload on `ctx.current_command` before returning the Command to the pool.** State Machine Contract §3.4 explicitly leaves this open: "the popped command is returned, so concrete states must read their target from elsewhere (e.g., ctx.current_command — to be defined when concrete states ship)." Wave 2 ships the definition: `Unit.current_command: Dictionary = {}` with shape `{ "kind": StringName, "payload": Dictionary }`. The `StateMachine.transition_to_next` writes a defensive `payload.duplicate()` copy (so pool re-rent doesn't race), then returns the Command to the pool, then transitions. `_clear_current_command()` runs on the empty-queue → Idle path so a stale payload never leaks into Idle's tick. UnitState_Moving's `enter` reads `ctx.current_command.payload.target` — the StateMachine framework remains the single dispatcher; states read off the stashed dictionary.

- **`UnitState_Moving._sim_tick` drives `MovementComponent._sim_tick(dt)` directly.** Until the MovementSystem phase coordinator lands (a LATER item per wave-1 retro: "MovementSystem phase coordinator wiring"), Moving IS the per-tick driver for the MovementComponent. When the coordinator ships, it iterates registered MovementComponents during the `movement` phase and calls their `_sim_tick`; Moving's `_sim_tick` drops the explicit drive-call and just polls `path_state` / `is_moving`. This is a one-line refactor and is documented in both the source code (`unit_state_moving.gd` block comment above `_sim_tick`) and in the BUILD_LOG wave-2 retro under LATER items.

- **Arrival detection uses an `_arrival_pending` latch flipped on first READY observation, NOT on first `is_moving == true` observation.** Initial design used `is_moving == true` as the latch trigger, but in single-tick-arrival cases (huge `move_speed`, short distance, e.g. the test scenario `move_speed=100, target=(1,0,0)`) the MovementComponent ingests waypoints, advances position, and sets `_waypoint_index = waypoints.size()` all within a single `_sim_tick` call — so Moving never observes `is_moving == true`. Switched to `path_state == READY` as the latch trigger. The latch closes the "have a path" race correctly: Moving's `enter()` issues `request_repath` but the scheduler is non-blocking, so on the entry tick `is_moving` is false because no waypoints exist yet. Without the latch we'd transition back to Idle on the entry tick, never having moved. The READY-as-latch trigger handles both the single-tick and multi-tick arrival cases uniformly. Tests `test_moving_transitions_to_idle_on_arrival` and `test_full_idle_moving_idle_cycle` pin both shapes.

- **`UnitState_Moving.exit` cancels the in-flight repath request even though `MovementComponent.request_repath` would also cancel it on a subsequent call.** Contract §3.5 specifies states must own their own teardown. The next state may not be another Moving (could be Attacking, Constructing, or Idle), and those states won't re-issue a repath that would shadow ours. Explicit cancel in exit is cheap and matches the contract's intent. Implementation: `_movement._scheduler.cancel_repath(_movement._request_id)` then `_movement._request_id = -1` so MovementComponent's next `_sim_tick` doesn't poll the cancelled request.

- **`UnitState_Idle._sim_tick` writes `MeshInstance3D.scale` directly inside the sim tick.** Sim Contract §1.5 allows visual side-effects in `_process` (UI off-tick), but States are RefCounted and have no `_process`. The pulse runs from `_sim_tick` because that's the only hook a State has, and `MeshInstance3D.scale` is render-only — not a SimNode field, doesn't trigger the on-tick assert (we don't route through `_set_sim`), doesn't feed into combat math, pathfinding, or any sim-state read. This is the same carve-out MovementComponent uses for `Node3D.global_position` (Sim Contract §4.1). The pulse is driven by `SimClock.tick * SIM_DT` — deterministic, replay-safe, and removes a wall-clock divergence vector from headless determinism tests even though we don't strictly need determinism for visual decoration.

- **`UnitState_Moving.interrupt_level = COMBAT`, NOT `NEVER`.** Contract §2.1 spec: NONE = damage doesn't interrupt, COMBAT = damage interrupts, NEVER = damage cannot interrupt. "Non-combat movement" sits in the COMBAT bucket per the contract's own examples. Phase 2's combat system can flip this to NONE for "casual" movement (e.g., hero traveling between bases) if combat balance demands it; one-line edit. Player `replace_command` always wins regardless of `interrupt_level` (Contract §3.5).

- **What did not ship in wave 2** (intentionally out of scope per kickoff): Concrete additional states (Attacking, Gathering, Constructing, Casting, Dying) — Phase 2+ when their owning systems exist. GroupMoveController / formation movement — Phase 1 session 2 (planned row in §2 unchanged). MovementSystem phase coordinator — LATER item, see below. F3 state-machine debug overlay — concrete overlays land WITH their owning systems per kickoff doc rule; visual ID readout for selected unit lands when the selected-unit panel ships (Phase 1 session 2). The 6 deliverable boxes from the wave-2 brief: UnitState_Idle ✅, UnitState_Moving ✅, Wire Idle/Moving in Unit base ✅, 12-18 tests ✅ (13 shipped: 4 Idle + 8 Moving + 1 base-class registration test), ARCHITECTURE.md update ✅ (this entry), BUILD_LOG.md wave-2 entry ✅.

### v0.14.4 — Phase 1 Session 2 wave 2A (2026-05-04) — ui-developer (Control groups + double-click-select-of-type)

- **`ControlGroups` is an autoload, ordered AFTER `SelectionManager`.** Control groups outlive any single scene transition (UI overlay, pause menu, debug panel) and need a single global instance accessible to any future hotkey rebinder, replay player, or panel UI. Mirrors the SelectionManager pattern. The `[autoload]` order matters at parse time — at `ControlGroups._ready`, `SelectionManager` must already be parsed so `SelectionManager.selected_units` returns live data. The kickoff explicitly specified this ordering.

- **Double-tap detection uses `SimClock.tick`, NOT wall-clock.** L5 lint forbids `Time.get_*_msec` / `OS.get_unix_time` in gameplay code; the project's replay-determinism guarantee depends on every timer reading the sim clock. 350ms (kickoff guess) at 30Hz is 10.5 ticks; we round to `DOUBLE_TAP_TICKS = 10`. The same constraint applies to `DoubleClickSelect` — `DOUBLE_CLICK_TICKS = 9` (~300ms). Both constants are knobs on the public class for easy tuning by lead during interactive smoke testing.

- **Double-click detection is signal-driven (kickoff option b), not input-event sibling-ordered (option a).** Both options are in the kickoff doc. Option (a) — sibling-order a second left-click handler after BoxSelectHandler — is fragile because BoxSelectHandler claims the left-press first via `set_input_as_handled`, and a second listener would have to coordinate with the box-select's click-vs-drag arbitration. Option (b) subscribes to `EventBus.selection_changed`: whenever the payload is exactly one unit (single-target select), compare against the previously-armed unit + tick. The signal-based approach is INDEPENDENT of which input layer routed the click — ClickHandler, BoxSelectHandler, or any future single-click producer all funnel through SelectionManager and emit the same broadcast. This makes the double-click detector a pure observer; it never touches raw mouse events.

- **Multi-select payload disarms the double-click tracker.** Without this, a sequence like "box-select 5 → click one of them → click that one again" would false-fire as a double-click on the second single-click (because the prior `_armed_unit_id` was set during a multi-select-then-single-select pattern). The fix: any selection_changed payload with size != 1 (whether multi-select OR an empty deselect_all) clears the armed state. Single-target selections are the only ones that arm or fire.

- **`center_on(Vector3)` added to CameraController as a single-line public method.** The kickoff explicitly authorized this scoped addition; not a refactor. Implementation: clamp the input position via the existing `clamp_to_bounds` helper (Y forced to 0 — the camera target rides on the ground plane), set `target_position`, and call `_apply_transforms()` so the change takes effect immediately. The clamp prevents the camera from snapping outside the map — important when the player double-taps a control group that's near the map edge. Animated centering (lerp toward target over a few frames) is a future polish; snap is the MVP.

- **No InputMap entries for Ctrl+1..9 / 1..9.** The control groups autoload reads keycodes directly off `InputEventKey` (via `event.keycode` and `event.ctrl_pressed`). Same pattern the camera controller uses for WASD (`Input.is_key_pressed(KEY_W)`). Adding nine InputMap actions per digit, each with a Ctrl-modifier variant, would be 18 entries for a 1:1 mapping with the digit-keycode. Direct keycode reads are simpler and survive the absence of an action registration without breaking.

- **`recall(n)` is a no-op on unbound or empty groups (does NOT clear selection).** The kickoff smoke-test sequence implies recall is a one-way restore: "Lead deselects, hits 1, sees the same 3 selected." If recall on an unbound group cleared the selection, the player's behavior would surprise them — pressing 5 for a group they never bound shouldn't lose their current selection. The defensive no-op matches the convention in StarCraft 2 and AoE2.

- **Centroid computation skips freed group members and ignores Y.** `_centroid_of_group(n)` filters via `is_instance_valid` (lazy pattern, mirrors SelectionManager) and computes mean XZ of live members. Y is set to 0 because the camera target_position rides on the ground plane (per `_apply_transforms`'s pitch math). Empty groups (post-filter) yield `Vector3.ZERO` which the caller treats as "no center fires" — defensive, doesn't drag the camera to origin spuriously.

- **`select_visible_of_type` always includes the target unit.** Defensive against a 1-frame projection edge case where the target's `is_position_behind` flickers true between click-time and signal-handler-time. The player's intent is clear: "select all of this type starting from THIS one." Hardcoding the target's inclusion makes the contract robust to projection noise.

- **Lazy-resolved CameraController via duck-typed scene walk.** `ControlGroups` doesn't `preload` `camera_controller.gd` because that would couple the autoload to a specific class file path. Instead, on first centering call, it walks `get_tree().current_scene` looking for a node with the unique CameraController surface (`has_method(&"center_on")` AND `target_position in node`). Same duck-typing pattern `box_select_handler.gd::_resolve_click_handler` uses for `process_left_click_hit`. The `set_camera_target(stub)` test seam injects a RefCounted stub.

- **37 new tests** (+23 in `test_control_groups.gd`, +14 in `test_double_click_select.gd`). Coverage: bind/recall correctness, freed-member filtering, double-tap timing window (in/out, same-key/cross-key), centroid math (single, multi, with-frees, empty), reset semantics, synthetic InputEventKey dispatch (press/release, echo, non-digit, Ctrl-vs-bare). ### v0.8.0 — Phase 0 Session 4 wave 2 (2026-05-01) — qa-engineer

- **`class_name MatchHarness` removed; callers use preloaded script ref.** The wave-2 kickoff specified `class_name MatchHarness extends RefCounted`. The Godot 4.6.2 class_name registry race (documented in §6 v0.4.0) bites RefCounted-based scripts the same as Node-based ones when GUT collects test files before the class_name registry populates. Removing the class_name declaration and switching to the established project pattern (preload the script, call `.new()`, then call `start_match()`) resolves the compile error. The Testing Contract §3.1 specified `static func new(seed, scenario) -> MatchHarness` as the factory shape; with class_name removed, a static factory can't reference its own type. Replaced with `func start_match(seed, scenario)` on a freshly `.new()`'d instance. Test callers define a `_make(seed, scenario)` helper to keep the line count low. No behavior change; contract behavior is preserved exactly.

- **`_test_set_farr` is off-tick, not on-tick as the Testing Contract worked example implied.** Testing Contract §3.3 showed `_test_set_farr` called inside `advance_ticks`. The initial linter-generated harness implemented this by connecting a one-shot lambda to `EventBus.sim_phase` and driving an internal `_test_run_tick()`. This caused "Cannot disconnect from 'sim_phase': callable is null" errors because GDScript closures and self-referential lambda disconnects interact badly in this context. Simplified to a direct off-tick write to `FarrSystem._farr_x100` (same pattern as `FarrSystem.reset()`) with a synthetic `EventBus.farr_changed` emit. The semantic is unchanged — test code sets Farr to a specific value without going through `apply_farr_change`'s on-tick assert. The side effect is `_test_set_farr` no longer advances `SimClock.tick` by 1; tests that need a specific tick count can be explicit about it. The Testing Contract §3.3 example note "must run inside a tick" is advisory context from the worked example, not a hard requirement for the method itself.

- **`FarrSystem.reset()` added cross-domain.** The wave-2 kickoff doc explicitly authorized this: "gameplay-systems' wave-1 retro flagged that reset() not added in this skeleton. You're MatchHarness — you need it. Add it (cross-domain, but small; document in §6)." Added `reset()` to `farr_system.gd` that reads the starting_value from BalanceData (same logic as `_load_starting_value_from_balance_data`), writes `_farr_x100`, and emits a synthetic `farr_changed` with reason `"harness_reset"`. The off-tick emit is intentional and documented in the source comment.

- **`MatchHarness.start_match` vs `_setup` naming.** The linter auto-generated `static func create(seed, scenario)` calling a private `_setup()`. After removing the static factory (class_name removal), renamed `_setup()` to be called from `start_match(seed, scenario)`. The public API name `start_match` better communicates intent than `create` — it's not constructing data; it's starting a match simulation.

- **`Scenarios.CATALOG` is a data-only dictionary, not a class with methods.** The initial Scenarios design had a static `apply(scenario_key, harness)` method. The linter simplified it to a `const CATALOG: Dictionary` with string keys → Dictionary of override values. MatchHarness._setup reads from the catalog dict and applies overrides directly. This is simpler and doesn't require Scenarios to know about the MatchHarness type. Adding a new scenario is a one-line Dict entry — no method needed.

- **`tests/integration/test_match_harness.gd` added alongside `tests/integration/test_determinism.gd`.** The linter created `test_match_harness.gd` in the integration directory in parallel. Both files exercise the harness: `test_match_harness.gd` covers the API surface (25 tests); `test_determinism.gd` covers the empty-match determinism guarantee (3 tests). The `tests/unit/test_match_harness.gd` tests overlap slightly — this is intentional redundancy for robustness. Test counts: 199 before wave 2 → 250 after (51 new tests; 1 intentional Pending remains from ui-developer's HUD defensive-fallback test).

- **What did not ship in wave 2 (intentionally out of scope per kickoff):** `spawn_resource_node` test helper (Phase 3). `MatchLogger` NDJSON writer (Phase 6). AI-vs-AI batch runner (Phase 6). `GameRNG` autoload wiring in harness (engine-architect; harness uses `seed()` on global RNG as fallback with TODO comment).
